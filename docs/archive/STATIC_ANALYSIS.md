# Static Analysis Tools

This project uses multiple static analysis tools to ensure code quality and catch potential bugs before they reach production.

## Available Tools

### C++ Static Analysis

#### 1. **cppcheck**

- **Purpose**: General-purpose C++ static analysis
- **Checks**: Warnings, performance, style, portability
- **Usage**: `scripts/run_linters.sh` (runs automatically)
- **Installation**: `brew install cppcheck`
- **Key Features**:
  - Focus on low false-positive rate by default
  - Can analyze C++ code (despite the name)
  - Supports parallel analysis with `-j` flag
  - Portability checks that other tools often miss

#### 2. **Clang Static Analyzer**

- **Purpose**: Deep inter-procedural analysis
- **Checks**: Memory safety, null pointer dereferences, use-after-free
- **Usage**: `scripts/run_linters.sh` (runs automatically)
- **Installation**: Comes with Xcode/Clang

#### 2b. **clang-tidy**

- **Purpose**: Additional linting and static analysis checks
- **Checks**: Performance, bug-prone patterns, readability, security
- **Usage**: Can be integrated into build system or run manually
- **Installation**: Comes with Clang/LLVM
- **Note**: Requires configuration to disable noisy checks (see [Best Practices](#best-practices))

#### 3. **Infer** (Facebook)

- **Purpose**: Advanced static analysis for memory safety and concurrency bugs
- **Checks**: Null pointer dereferences, resource leaks, race conditions
- **Usage**: `scripts/run_linters.sh` (runs automatically)
- **Installation**: `brew install infer` or see [Installation Guide](#infer-installation)
- **Website**: <https://fbinfer.com/>

### Python Security

#### **Bandit**

- **Purpose**: Security-focused static analysis for Python
- **Checks**: Security vulnerabilities, common mistakes
- **Usage**: `scripts/run_linters.sh` (runs automatically)
- **Installation**: `pip install bandit` or use `.bandit-env` virtual environment

### Other Languages

- **Swift**: `swiftlint` (for macOS app)
- **C++ TUI**: Uses same C++ static analysis tools (cppcheck, clang-analyzer, infer)

## Running Static Analysis

### Run All Linters

```bash
./scripts/run_linters.sh
```

This runs all available static analysis tools. Tools that aren't installed will be skipped with a warning.

### Run Individual Tools

#### cppcheck

```bash
cppcheck --enable=warning,performance,style,portability \
  --std=c++17 \
  --suppress=missingIncludeSystem \
  native/src native/include native/tests
```

#### Clang Static Analyzer

```bash

# Ensure compile_commands.json exists

cmake --preset macos-x86_64-debug
cd build/macos-x86_64-debug
clang++ --analyze -std=c++20 native/src/*.cpp
```

#### Infer

```bash

# Ensure compile_commands.json exists

cmake --preset macos-x86_64-debug
cd build/macos-x86_64-debug
infer run --compilation-database compile_commands.json
```

## Infer Installation

**Official Guide**: See [Infer Getting Started](https://fbinfer.com/docs/getting-started) for the most up-to-date installation instructions.

### Option 1: Binary Release (Recommended)

Download the latest binary release from [GitHub Releases](https://github.com/facebook/infer/releases). For macOS:

```bash

# Replace VERSION with latest release (e.g., VERSION=1.2.0)

VERSION=1.2.0
curl -sSL "https://github.com/facebook/infer/releases/download/v${VERSION}/infer-macos64-v${VERSION}.tar.xz" \
  | tar -C /opt -xJ && \
  sudo ln -s "/opt/infer-macos64-v${VERSION}/bin/infer" /usr/local/bin/infer
```

Or install to a local directory:

```bash
VERSION=1.2.0
mkdir -p ~/bin
curl -sSL "https://github.com/facebook/infer/releases/download/v${VERSION}/infer-macos64-v${VERSION}.tar.xz" \
  | tar -C ~/bin -xJ
export PATH="$HOME/bin/infer-macos64-v${VERSION}/bin:$PATH"
```

### Option 2: Homebrew (if available)

```bash
brew install infer
```

**Note**: Homebrew may not always have the latest version. Check [Homebrew Formula](https://formulae.brew.sh/formula/infer) for availability.

### Option 3: Build from Source

Infer is written in OCaml and requires OCaml, opam, and other dependencies. See the [official installation
guide](https://fbinfer.com/docs/getting-started) for detailed instructions.

Quick install from source:

```bash

# Install dependencies

brew install opam
opam init
opam switch create 4.14.0
eval $(opam env)

# Clone and build Infer

git clone https://github.com/facebook/infer.git
cd infer
./build-infer.sh
export PATH="$PWD/infer/bin:$PATH"
```

### Option 4: Docker

Use Infer via Docker for consistent environment:

```bash
docker pull infer/infer:latest

# Run Infer in Docker

docker run --rm -v "$(pwd):/workspace" -w /workspace \
  infer/infer:latest \
  infer run --compilation-database compile_commands.json
```

### Verify Installation

```bash
infer --version
```

You should see the Infer version number if installed correctly.

## Infer Usage

### Basic Usage

Infer works best with `compile_commands.json`, which is automatically generated by CMake:

```bash

# Configure project (generates compile_commands.json)

cmake --preset macos-x86_64-debug

# Run Infer

cd build/macos-x86_64-debug
infer run --compilation-database compile_commands.json
```

### Viewing Results

Infer generates results in `infer-out/` directory:

```bash

# View HTML report

infer-explore --html infer-out

# View JSON report

cat infer-out/report.json | jq

# View specific bug types

infer-explore --html infer-out --select <bug-type>
```

### Common Bug Types

Infer detects various bug types:

- **NULL_DEREFERENCE**: Null pointer dereference
- **RESOURCE_LEAK**: Memory or resource leak
- **USE_AFTER_FREE**: Use after free
- **RACE_CONDITION**: Concurrency issues
- **DEADLOCK**: Potential deadlock
- **INFINITE_LOOP**: Infinite loop detection

### Filtering Results

```bash

# Only show high-severity bugs

infer-explore --html infer-out --only-show

# Filter by file

infer-explore --html infer-out --select native/src/box_spread_strategy.cpp
```

## Integration with CI/CD

The `scripts/run_linters.sh` script is designed to be used in CI/CD pipelines:

```yaml

# Example GitHub Actions
- name: Run Static Analysis
  run: ./scripts/run_linters.sh
```

The script:

- Exits with non-zero code if any tool finds issues
- Skips tools that aren't installed (with warnings)
- Provides clear error messages

```
--suppress=missingIncludeSystem
--suppress=unusedFunction
```

### Infer

Create `.inferconfig` in project root:

```json
{
  "infer-args": ["--no-progress-bar", "--quiet"]
}
```

### Clang Static Analyzer

Configure via CMake flags or `.clang-tidy` file.

## Handling Legacy Code

When introducing static analysis to an existing codebase, you may face thousands of warnings from legacy code. Rather than fixing everything at once
(which is impractical), you can use a **mass suppression** strategy:

### Strategy: Suppress Legacy, Focus on New Code

1. **Initial Suppression**: Suppress all existing warnings in legacy code
2. **Track New Issues**: Only show warnings from new or modified code
3. **Gradual Cleanup**: Fix suppressed warnings over time as you touch those files

This approach allows you to:

- Start benefiting from static analysis immediately
- Avoid overwhelming developers with thousands of legacy warnings
- Focus on preventing new bugs
- Gradually improve code quality over time

### Implementation

Different tools support this in different ways:

- **cppcheck**: Use `--suppressions-list=suppressions.txt` to suppress specific warnings
- **Clang Static Analyzer**: Use `// NOLINT` comments or suppression files
- **Infer**: Supports suppression files and can filter by file modification time
- **PVS-Studio**: Has built-in mass suppression mechanism (see [reference article](https://pvs-studio.com/en/blog/posts/0527/))

### Best Practices for Suppression

1. **Document Suppressions**: Always document why a warning is suppressed
2. **Review Periodically**: Periodically review suppressed warnings to see if they can be fixed
3. **Don't Suppress New Code**: Never suppress warnings in new code - fix them instead
4. **Use Tool-Specific Mechanisms**: Prefer tool-specific suppression mechanisms over broad suppressions

## Best Practices

1. **Run before commits**: Use `scripts/run_linters.sh` before committing
2. **Fix high-severity issues first**: Address critical bugs before style issues
3. **Review false positives**: Some tools may report false positives; suppress them appropriately
4. **Keep tools updated**: Update static analysis tools regularly for better detection
5. **Use in CI**: Integrate static analysis into your CI/CD pipeline
6. **Handle legacy code strategically**: Use mass suppression for legacy code, focus on new code

## Troubleshooting

### compile_commands.json Not Found

```bash

# Ensure CMake is configured with compile commands export

cmake --preset macos-x86_64-debug

# Or manually:

cmake -S . -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
```

### Infer Fails to Analyze

- Ensure `compile_commands.json` uses `arguments` field (not `command`)
- Check that all source files compile successfully
- Try cleaning and rebuilding: `rm -rf build && cmake --preset macos-x86_64-debug`

### Too Many False Positives

- Use tool-specific suppression mechanisms
- Review and refine suppressions regularly
- Consider tool configuration files (`.cppcheckrc`, `.inferconfig`)
